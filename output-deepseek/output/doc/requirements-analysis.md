# 软件台账管理平台需求分析文档

## 需求背景
1. 为了对公司使用的软件资产进行统一管理，建立软件台账管理平台，提升软件资产的管控能力。
2. 通过平台实现软件台账和软件使用关系台账的集中管理，支持多维度查询、筛选和统计，提高软件资产的可视化水平。
3. 满足不同角色对软件数据查看和管理的需求，确保数据安全性和权限控制。
4. 提供可观测能力，包括数据统计大屏和同步状态监控，便于运维和管理决策。

## 需求要点
1. **软件台账管理**：包含软件基本信息管理和版本介质管理，支持产品类、组件类两种管理类型。
2. **软件使用关系台账管理**：分为终端类、应用系统类、服务器三大类，每类下又细分小类别，记录软件使用关系及合规状态。
3. **可观测需求**：提供软件数据统计大屏和同步情况监控大屏，支持多维度统计和告警下钻。
4. **权限需求**：根据不同角色（超管、团队负责人、普通用户）控制数据查看范围。

## 需求详细描述

### 1. 软件台账

#### 1.1 软件数据模型
- **软件模型字段**：
  - 软件ID（唯一标识）
  - 软件名称
  - 软件管理类型：商业软件、开源软件、自研软件
  - 软件功能类型（一级）：办公、开发测试、运维管理、系统类等
  - 软件功能类型（二级）：如办公类-文字处理、办公类-多媒体处理等
  - 软件管理类型（介质类型）：产品类、组件类
  - 软件管理类型（仓库类型，仅组件类）：maven、npm、pypi等
  - 软件负责人
  - 开源协议（仅开源软件）
  - 使用限制：使用限制范围
  - 软件功能介绍
  - 软件使用手册

- **软件版本模型字段**：
  - 管理的软件ID
  - 软件版本
  - 介质名称
  - 介质下载地址
  - 开源协议（仅开源软件）
  - 介质描述
  - 介质类型：安装包、镜像、SDK等
  - 介质大小
  - 使用限制：使用限制范围
  - 使用说明：应用系统、终端

#### 1.2 数据来源
- 部分数据来源于引入流程（人工录入或导入）
- 部分数据来源于其他系统同步（需在同步检测中展示）

#### 1.3 页面
- **列表页**：展示软件基本信息和版本信息，支持多维度筛选、搜索、排序。
- **详情页**：展示软件各个版本介质的详细情况，包括下载地址、介质类型、大小，支持筛选、排序和搜索。

#### 1.4 权限
- 软件台账数据为非敏感数据，无需登录均可查看。

### 2. 软件使用关系台账

#### 2.1 数据模型
- **终端使用关系数据模型（A1）**：数据量约2000万条
  - 终端MAC（关联终端设备数据）
  - 终端IP
  - 终端使用软件名称
  - 终端使用软件版本
  - 软件是否合规
  - 软件是否为黑名单软件
  - 终端使用软件安装路径
  - 终端使用软件安装时间
  - 数据状态：正常、残留数据、未知

- **终端设备数据模型（A2）**：数据量约10万条，一条A2记录对应多条A1记录
  - 终端MAC
  - 终端IP
  - 终端网段：办公、开发测试、生产
  - 终端名称
  - 终端负责人
  - 终端负责人团队
  - 终端状态：正常、已回收、未知

- **应用系统使用关系数据模型-组件类（B1）**：数据量约1000万条
  - 应用系统名称
  - 应用系统使用软件版本
  - 应用系统使用软件名称
  - 软件是否合规
  - 软件是否为黑名单软件
  - 应用系统使用软件管理类型（介质类型）：产品类、组件类
  - 应用系统使用软件管理类型（仓库类型，仅组件类）：maven、npm、pypi等
  - 应用系统使用软件开源协议

- **应用系统使用关系数据模型-产品类（B2）**：数据量约1000万条
  - 应用系统名称
  - 应用系统使用软件版本
  - 应用系统使用软件名称
  - 软件是否合规
  - 软件是否为黑名单软件
  - 所在服务器IP
  - 应用系统使用软件管理类型（介质类型）：产品类、组件类
  - 应用系统使用软件开源协议

- **应用系统数据模型（B3）**：数据量约1万条
  - 应用系统名称（与B1、B2关联）
  - 应用系统负责人
  - 应用系统团队负责人
  - 应用系统类型
  - 应用系统级别

- **服务器使用关系数据模型（C1）**：数据量较大，按软件类型分表存储（pypi约500万、pkg约2000万、jar约2000万）
  - 服务器IP
  - 服务器名称
  - 服务器类型
  - 安装软件名称
  - 安装软件版本
  - 软件是否合规
  - 软件是否为黑名单软件
  - 安装软件位置
  - 安装软件类型（pypi、pkg、jar等，每种类型一张表）
  - 其他字段（根据软件类型的不同有不同的字段）

- **服务器数据模型（C2）**
  - 服务器IP（与C1关联）
  - 服务器所属应用系统（与B3关联）

#### 2.2 数据来源
- 终端数据来源于终端管理平台
- 应用系统数据来源于应用系统管理平台和DevOps平台
  - DevOps平台数据主要是组件类软件使用关系台账
  - 应用系统管理平台数据主要是产品类软件使用关系台账
- 服务器数据来源于服务器管理平台

#### 2.3 页面
- **终端使用关系列表页**：展示终端使用关系，关联到负责人，支持多维度筛选、搜索、排序。
- **应用系统使用关系列表页**：展示应用系统使用关系，关联到负责人，支持多维度筛选、搜索、排序。
- **服务器使用关系列表页**：展示服务器使用关系，关联到负责人，支持多维度筛选、搜索、排序。

#### 2.4 权限
- **终端使用关系台账**：终端负责人本人、超管、终端负责人所属团队负责人可查看。
- **应用系统使用关系台账**：应用系统负责人、超管、应用系统所属团队负责人可查看。
- **服务器使用关系台账**：服务器所属应用系统负责人、超管、应用系统所属团队负责人可查看。

### 3. 可观测需求

#### 3.1 软件数据统计大屏
- **软件使用关系统计**（所有人可查看）：
  - 按照终端、服务器、应用系统三个使用场景的使用数据统计
  - 每个使用场景下细致的分类统计（团队负责人和超管可查看）
    - 终端：使用网段统计、终端所属团队统计、软件管理类型统计（开源软件、商业软件、自研软件）
    - 应用系统：软件类型统计（产品类、组件类，组件类继续下钻maven、pypi等）、应用系统所属团队统计、软件管理类型统计、应用系统级别统计
    - 服务器：软件类型统计（按照工具采集的类型统计，pkg、jar、pypi等）、软件管理类型统计、服务器所属应用系统所属团队统计、应用系统级别统计
- **软件台账统计**（所有人可查看）：
  - 统计维度1：产品类、组件类
  - 统计维度2：开源软件、商业软件、自研软件
  - 统计维度3：按照功能类型统计：办公类、开发类、运维管理类、安全类等

#### 3.2 同步情况
- **同步状态统计**：统计同步状态，包括同步中、同步成功、同步失败等（仅超管可查看）。
- **同步异常情况检测**：针对同步过程出现明显报错或者前后两次同步明显数据不一致的情况进行大屏展示、告警、详情下钻展示。
- **同步失败详情查看**：统计同步失败原因，包括同步失败次数、同步失败原因等。
- **数据变动检测**：检测数据的新增、修改、删除变更数据的统计分析。

## 非功能性需求

### 性能要求
- **并发要求**：支持至少100并发用户同时操作。
- **响应时间**：列表页加载时间不超过3秒，详情页加载时间不超过5秒。
- **数据量**：支持千万级数据表的查询和筛选，需考虑分页、索引优化。

### 可用性要求
- 系统可用性达到99.9%。
- 支持灰度发布和回滚机制。

### 数据一致性要求
- 数据同步过程保证最终一致性，同步失败需有重试和告警机制。
- 关键数据操作需记录操作日志。

### 可观测要求
- **监控要求**：系统关键指标监控（CPU、内存、磁盘、网络）、业务指标监控（同步成功率、数据量变化）。
- **统计分析要求**：提供灵活的统计图表，支持自定义时间段和多维度分析。
- **告警要求**：同步失败、系统异常等关键事件实时告警，支持邮件、钉钉等多种通知方式。

### 数据规模分析与技术要求
- **终端使用关系数据**：约2000万条，需考虑分库分表或读写分离。
- **应用系统使用关系数据**：约2000万条（组件类+产品类），需按类型分表。
- **服务器使用关系数据**：约4500万条（按软件类型分表），需分区存储。
- **技术要求**：后端需支持高并发查询、缓存机制（Redis）、异步任务处理（Celery或类似）。前端需支持大数据量表格渲染、图表可视化（ECharts等）。

## 待澄清问题
1. **技术栈偏好**：前端、后端、数据库等技术栈是否有特定要求？
2. **UI/UX风格**：是否有现有的设计系统或风格规范需要遵循？
3. **集成系统**：需要同步的其他系统的接口文档和数据格式是否已提供？
4. **部署环境**：预计部署在公有云还是私有云？是否有容器化要求？
5. **安全要求**：除角色权限外，是否有其他安全要求（如数据加密、审计日志）？
6. **时间要求**：项目期望的里程碑和交付时间点？

## 假设与约束
- 假设用户对软件台账数据的查看无需登录，但使用关系台账需要登录验证。
- 假设数据同步通过定时任务或消息队列实现，具体技术选型待定。
- 假设大屏展示使用独立的可视化组件，支持实时刷新。
- 约束：数据量庞大，需在数据库设计和查询优化上投入精力。

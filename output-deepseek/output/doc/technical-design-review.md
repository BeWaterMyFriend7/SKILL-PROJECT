# 技术方案评审报告

## 评审基本信息
- **评审对象**：软件台账管理平台技术方案设计
- **评审版本**：v1.0
- **评审日期**：2025-02-24
- **评审人员**：架构师、技术负责人、DBA、安全专家

## 总体评价
技术方案整体设计合理，覆盖了需求要点，架构清晰，数据库设计考虑了数据量大的特点，接口设计完备，非功能设计有针对性。方案具备可实施性。

## 潜在问题与优化建议

### 1. 数据库设计方面
#### 问题
- **终端使用关系表分区策略**：按 `terminal_mac` 哈希分区可能导致数据倾斜（某些MAC对应大量记录），影响查询性能。
- **服务器使用关系分表**：按软件类型（pkg、jar、pypi）分表，但未来新增软件类型需要修改表结构，扩展性不足。
- **JSON字段使用**：`server_usage_*` 表中的 `extra_fields` 使用 JSON 类型，虽然灵活但不利于复杂查询和索引。

#### 建议
1. 终端使用关系表改为按 `created_at` 时间范围分区（按月或按季度），结合 `terminal_mac` 二级哈希分区，平衡负载。
2. 服务器使用关系考虑使用通用表结构，增加 `software_type` 字段区分类型，配合分区或分片，避免动态建表。
3. JSON 字段保留，但关键字段应提取为单独列，并建立索引；JSON 仅用于存储非关键扩展信息。

### 2. 接口设计方面
#### 问题
- **使用关系查询接口权限校验**：每个接口都需要根据角色动态过滤数据，权限逻辑复杂，可能影响性能。
- **统计报表接口数据实时性**：大屏要求实时数据，但统计计算可能较慢，影响用户体验。

#### 建议
1. 权限校验建议在网关层统一处理，通过用户角色和所属团队信息注入查询条件，减少业务层重复代码。
2. 统计报表采用预计算 + 增量更新策略：建立统计结果表，定时（如每5分钟）更新，接口直接查询结果表；实时性要求极高的数据可采用流式计算（如 Flink）。

### 3. 非功能设计方面
#### 问题
- **缓存策略较简单**：仅提到缓存统计结果，对于高频查询的软件列表、详情等未设计缓存。
- **同步服务可靠性**：外部系统接口不稳定时，重试机制和补偿事务需要详细设计。

#### 建议
1. 增加多级缓存：热点软件信息缓存到 Redis（TTL 1小时），使用 LRU 策略；列表查询结果可缓存短时间（TTL 1分钟）。
2. 同步服务引入可靠性模式：
   - 重试策略：指数退避重试，最多3次。
   - 死信队列：失败消息进入死信队列，人工干预。
   - 数据一致性：采用最终一致性，记录同步流水，支持数据修复。

### 4. 技术选型方面
#### 问题
- **消息队列选型未明确**：RabbitMQ 和 Kafka 特性不同，需要根据场景选择。
- **前端框架 Vue 3 学习成本**：团队若原有 React 技术栈，切换成本需评估。

#### 建议
1. 消息队列推荐 Kafka，因其高吞吐、持久化、流处理能力更适合大数据同步场景。
2. 前端框架尊重团队现有技术栈，若团队熟悉 React，可改用 React + Ant Design，减少学习成本。

### 5. 安全性方面
#### 问题
- **敏感数据传输**：仅提到 HTTPS，未提及 API 防篡改、防重放机制。
- **数据库字段加密**：哪些字段需要加密未明确。

#### 建议
1. API 增加签名验证（如 HMAC）或使用 OAuth 2.0，防止未授权调用。
2. 明确加密字段：用户密码（如有）、API密钥、个人身份证号等；软件下载地址等非敏感字段无需加密。

### 6. 可维护性方面
#### 问题
- **微服务拆分边界**：方案中未明确服务拆分粒度，可能导致单体臃肿或微服务过细。

#### 建议
1. 建议按领域拆分服务：软件台账服务、使用关系服务、同步服务、统计服务、权限服务。每个服务独立数据库（但需考虑分布式事务）。
2. 或采用模块化单体架构，通过包结构隔离，未来可按需拆分。

### 7. 部署与运维方面
#### 问题
- **监控告警阈值未定义**：如同步失败率多少触发告警、响应时间阈值等。

#### 建议
1. 定义明确监控指标与告警阈值：
   - 同步失败率 > 5% 触发警告，> 10% 触发严重告警。
   - API 平均响应时间 > 3s 告警，> 10s 严重告警。
   - 系统可用性 < 99.9% 告警。

## 风险评估与应对

| 风险项 | 影响程度 | 发生概率 | 应对措施 |
|--------|----------|----------|----------|
| 外部系统接口变化 | 高 | 中 | 设计适配器模式，接口变更时只需修改适配器；建立接口契约测试。 |
| 数据量增长超出预期 | 高 | 低 | 定期评估数据增长，提前规划分库分表；预留扩容方案。 |
| 团队技术栈不匹配 | 中 | 中 | 技术选型前调研团队技能；安排培训或引入外部专家。 |
| 安全漏洞 | 高 | 低 | 定期安全扫描、渗透测试；建立安全开发规范。 |
| 项目延期 | 中 | 中 | 采用敏捷迭代，每两周交付可运行版本；关键路径任务优先。 |

## 评审结论
**通过，但有条件**：技术方案总体可行，需针对上述问题在详细设计阶段进行优化。建议在下一阶段（详细设计）中完善以下内容：

1. 数据库详细分区方案、索引设计、SQL 性能测试。
2. 权限校验详细流程图与代码实现方案。
3. 缓存与同步可靠性详细设计。
4. 安全方案详细设计（API 安全、数据加密、审计日志）。

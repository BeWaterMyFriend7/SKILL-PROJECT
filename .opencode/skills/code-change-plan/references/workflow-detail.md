## 4.1 建立项目分析基线

收集并记录：

- 仓库根目录
- 当前分支
- 当前 HEAD
- 已修改和未跟踪文件
- 当前生效的 `AGENTS.md`
- 项目目录和模块边界
- 构建命令
- 格式化命令
- Lint 命令
- 类型检查命令
- 单元测试、集成测试和回归测试命令
- 代码生成命令
- 重要部署、迁移和回滚命令

规则：
- 不得覆盖用户已有的未提交修改
- 必须区分已有修改和本次计划修改
- 如果无法确定分支或 HEAD，需要在方案中明确说明
- 在方案中记录分析时的项目基线

## 4.2 归一化用户需求

将用户需求整理为：

- 业务目标：一句话描述用户想要达成的业务结果
- 用户可见结果：用户能看到什么变化
- 功能需求：以 REQ-xx 编号，MUST/SHOULD/NICE 优先级
- 非功能需求：性能、安全、稳定性、兼容性等
- 非目标：明确本次不做的事情

## 4.3 分析当前实现

对每个相关功能：

- 找到功能入口（API、CLI、事件触发等）
- 追踪调用链：入口 → 中间层 → 底层实现
- 识别数据模型和持久化方式
- 检查配置项和 Feature Flag
- 检查缓存、事件和消息机制
- 检查日志、指标和告警
- 确认单元测试、集成测试和测试数据

## 4.7 拆分为修改单元

每个修改单元必须包含：

| 字段 | 内容 |
|------|------|
| 目标 | 一句话 |
| 关联需求 | REQ-xx |
| 关联验收标准 | AC-xx |
| 前置依赖 | 必须先完成的 CU |

## 4.8 进行全项目影响分析

对每个修改单元，分析影响范围：

| ID | 影响项 | 文件/位置 | 影响类型 | 优先级 | 关联 CU |
|----|--------|----------|---------|--------|---------|
| | | | | MUST_CHANGE / SHOULD_CHANGE / OPTIONAL_CHANGE | |

覆盖范围必须包含：
- 功能入口
- 调用方和被调用方
- 接口和实现
- 数据模型和持久化
- 缓存、事件和消息
- 配置项和 Feature Flag
- 构建、部署、迁移和回滚脚本
- 单元测试、集成测试、测试数据和 Mock
- README、接口文档、设计文档和运维手册
- 日志、指标、链路追踪和告警

## 4.9 建立旧逻辑退役矩阵

对每个被替换的旧资产：

| ID | 关联 CU | 文件路径 | 符号 | 退役动作 | 验证项 |
|----|--------|---------|------|---------|--------|
| LR-01 | CU-01 | | | 删除 / 替换 / 迁移 / 限期兼容 | V-xx |

退役动作含义：
- **删除**：直接移除，无遗留引用
- **替换**：用新逻辑替代，旧代码移除
- **迁移**：数据或配置迁移后移除
- **限期兼容**：保留一段时间后移除，需标注截止条件

## 4.14 写入执行契约

审批后，执行 Codex 必须遵守：

1. 完整阅读本文档后再修改代码
2. 执行前核对审批状态、分支、HEAD 和工作区基线
3. 严格按照实施顺序执行
4. 只修改文件变更清单中的文件
5. 严格完成旧逻辑退役矩阵
6. 不得保留未经批准的新旧双逻辑
7. 每个修改单元完成后运行对应局部验证
8. 完成后运行全部标记为必需的验证项
9. 发现计划外影响或新的设计决策时立即停止
10. 默认不执行 `git add`、`git commit`、`git push`、破坏性 reset 或历史重写
11. 最终报告修改文件、旧逻辑清理、验证结果、未运行项、方案偏差和残余风险

### 必须停止并申请修订的情况

出现以下任意情况，状态标记为 `PLAN_AMENDMENT_REQUIRED`：

- 必须修改文件清单之外的文件
- 新发现接口、Schema、配置或数据迁移变化
- 发现退役矩阵遗漏的旧资产
- 方案中的指令互相冲突
- 测试证明方案定义的业务行为不正确
- 安全、权限、兼容或性能假设失效
- 当前分支、HEAD 或工作区基线已漂移

### 禁止事项

- 不得自行扩大需求范围
- 不得自行改变已经批准的业务决策
- 不得保留无明确用途的旧代码
- 不得新增未经批准的临时兼容层
- 不得通过跳过测试来声明完成
- 不得将"命令未运行"视为"验证通过"

## 4.15 定义完成状态

### VERIFIED
仅当以下条件全部满足：
- 所有批准的修改单元完成
- 所有必需验证通过
- 所有旧逻辑退役项关闭
- 没有计划外文件修改
- 最终 Diff 审核通过

### IMPLEMENTED_NOT_FULLY_VERIFIED
代码修改完成，但存在无法运行的必需验证。必须列出未运行项和原因，不得声明完全验证。

### BLOCKED
由于技术错误、环境问题或外部依赖无法继续。

### PLAN_AMENDMENT_REQUIRED
原方案不完整或必须调整。

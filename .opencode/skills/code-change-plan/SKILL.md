# Code Change Plan

将用户提出的代码修改需求，转换为一份完整、可评审、可直接交给 Codex 执行的 Markdown 变更方案。

> 标准工作流程详见 `references/workflow-detail.md`，报告结构详见 `references/report-structure.md`，质量检查详见 `references/quality-checklist.md`。

## 1. Skill 目标

最终方案必须明确说明：

- 用户真正需要解决的问题
- 当前项目中的实现方式
- 目标实现方式
- 修改前后的流程、调用链、数据流和配置流变化
- 涉及的代码、测试、脚本、配置、数据、接口和文档
- 哪些旧逻辑需要删除、替换、迁移或限期兼容
- Codex 后续应如何实施
- 如何测试、检查和验证
- 什么情况下必须停止执行并重新调整方案

本 Skill 只负责分析和生成方案，不负责修改业务代码。

## 2. 强制工作模式

始终以以下模式运行：

```text
PLAN_ONLY
```

仅允许写入：

```text
.codex/change-plans/<plan-id>.md
```

不得修改：生产代码、测试代码、配置文件、构建和部署脚本、数据库脚本、项目文档、生成文件、依赖文件。

只生成一个 Markdown 方案文件，不生成单独的 JSON 文件。

方案模板：
```text
assets/change-plan-template.md
```

## 3. 核心原则

### 3.1 先分析，再设计

不得根据用户的一句话直接生成通用修改方案。必须先检查项目现状，理解当前实现、调用关系和关联资产，再产出方案。

### 3.2 必须进行全项目影响分析

不得只检查用户点名的文件。至少需要追踪：功能入口、调用方和被调用方、接口和实现、数据模型和持久化、缓存、事件和消息、配置项和 Feature Flag、构建、部署、迁移和回滚脚本、单元测试、集成测试、测试数据和 Mock、README、接口文档、设计文档和运维手册、日志、指标、链路追踪和告警。

### 3.3 被替换的旧逻辑必须闭环处理

当目标方案替换当前实现时，必须识别所有会失效的旧资产。最终实现不得出现未设计的新旧逻辑同时生效。

### 3.4 方案必须可以直接执行

方案必须具体到：文件、类、函数、接口或配置项、修改动作、实施顺序、删除内容、验证命令、期望结果、停止条件。不得要求后续 Codex 重新分析主要设计问题。

### 3.5 所有结论必须可验证

每个需求、修改单元、影响项和旧逻辑退役项，都必须映射到明确的验证方式。


## 3.6 复杂度分级

根据变更范围选择模式：

| 模式 | 适用场景 | 章节数 |
|------|---------|--------|
| `SIMPLE` | 单文件修改、配置调整、小范围修复 | 精简版（合并相邻章节） |
| `FULL` | 多模块重构、接口变更、数据迁移 | 完整 23 章（默认） |

判断标准：涉及文件 ≤3 且无接口/数据/配置流变化 → SIMPLE，否则 FULL。

SIMPLE 模式的章节映射见 `references/report-structure.md`。

## 4. 标准工作流程概览

详细步骤和每步的执行规范见 `references/workflow-detail.md`。

1. 建立项目分析基线
2. 归一化用户需求
3. 分析当前实现
4. 识别现状与目标之间的差距
5. 设计目标方案
6. 描述修改前后变化
7. 拆分为修改单元
8. 进行全项目影响分析
9. 建立旧逻辑退役矩阵
10. 定义文件变更清单
11. 定义实施顺序
12. 定义测试和验证方案
13. 定义发布、回滚和风险
14. 写入执行契约
15. 定义完成状态

## 5. 报告结构

最终 Markdown 必须包含 23 个章节（详见 `references/report-structure.md`），不得省略。确实不适用时写"不适用：<原因>"。

## 6. 追踪关系要求

必须形成以下追踪关系：

```text
REQ -> AC -> CU -> 文件 -> V
```

同时形成：

```text
LR -> CU -> V
```

出现以下情况时，方案不完整：

- 需求没有验收标准
- 验收标准没有修改单元或验证项
- `MUST_CHANGE` 影响项没有修改单元
- 旧逻辑退役项没有验证
- 修改文件没有对应修改单元
- 必需验证项没有期望结果

## 7. 审批规则

初始状态必须是：

```text
WAITING_FOR_APPROVAL
```

Skill 不得自行批准方案。

审批记录必须包含：状态、审批范围、审批人、审批时间、审批意见。

允许状态：

```text
WAITING_FOR_APPROVAL
APPROVED
APPROVED_WITH_CHANGES
REJECTED
SUPERSEDED
```

## 8. 方案生成前的质量检查

输出最终方案前必须逐项核对。完整清单见 `references/quality-checklist.md`。

核心检查项：
- 是否准确表达用户需求
- 当前流程和目标流程是否清晰区分
- 是否使用准确项目路径
- 每个被替换的旧资产是否有处理方式
- 是否可能意外保留新旧双逻辑
- 文件动作是否明确
- 实施顺序是否合理
- 验证命令是否适用于当前项目
- 追踪关系是否完整
- 审批状态是否为 `WAITING_FOR_APPROVAL`

任何一项不满足，都应继续完善方案。

## 9. 最终回复要求

最终只需简要返回：

- Plan ID
- 方案文件路径
- 方案摘要
- 最高风险
- 需要人工决策的事项
- 明确说明本次没有修改业务代码

不得在同一次执行中实施方案。


## 10. 增量修订模式

当方案审批结果为 `APPROVED_WITH_CHANGES` 或执行中发现 `PLAN_AMENDMENT_REQUIRED` 时，使用增量修订代替全量重新生成。

### 触发条件
- 审批人标记 `APPROVED_WITH_CHANGES` 并指明修改范围
- 执行中发现 PLAN_AMENDMENT_REQUIRED（接口/Schema/配置遗漏等）

### 修订规则
- 保留原 Plan ID 不变
- 仅更新变更涉及的章节
- 保留已批准的修改单元，新增或调整部分标注为 `AMENDED`
- 修订后状态重置为 `WAITING_FOR_APPROVAL`
- 审批记录追加新行，标记修订版本号（V2, V3...）

### 修订模板
见 `assets/change-plan-amendment.md`。

### 修订提示词
`$code-change-plan 对 .codex/change-plans/<plan-id>.md 进行增量修订：<具体变更内容>`

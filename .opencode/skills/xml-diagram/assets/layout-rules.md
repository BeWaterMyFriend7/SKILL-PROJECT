# 布局规则

布局必须由模板和算法共同决定，不允许自由散点摆放。

## 通用安全区

```yaml
页面边距: 32
区域纵向间距: 36
区域横向间距: 36
区域标题高度: 44
区域左右内边距: 28
区域上下内边距: 24
卡片横向间距: 28
卡片纵向间距: 28
标签横向间距: 8
标签纵向间距: 8
任意元素最小间距: 24
```

## 必检规则

```text
区域标题与卡片不重叠。
卡片与卡片不重叠。
标签与标签不重叠。
卡片不超出区域。
箭头标签不遮挡节点。
图例不遮挡主图。
节点 y 必须大于 区域 y + 标题高度 + 上内边距。
```

## 画布利用率

```text
目标利用率：60%-80%。
低于 50%：缩小画布或重新居中内容。
高于 85%：扩展画布或换行。
图例和脚注必须计入内容范围。
```

画布修正顺序：

```text
1. 先看内容密度：节点、标签、区域是否足够表达主题。
2. 内容不足时先补语义，不要先缩画布。
3. 内容足够但布局稀疏时，重新分配网格、缩小区域间距或调整列数。
4. 内容足够且仍低于 50% 时，再缩小画布。
5. 内容过满时，优先换行、分组或增加画布，不要压缩标签间距。
```

## 内容充实预检

```text
生成架构图前先评估语义密度，而不是只评估画布大小。
如果用户输入只给出一个系统名或少量模块，先头脑风暴补齐候选内容。
候选维度包括：入口/用户、核心业务能力、服务模块、数据对象、外部系统、治理能力、监控运维、安全合规、部署资源。
通用技术内容可以直接作为合理假设写入图中；涉及业务事实、组织职责或领域边界不确定时，先向用户确认。
不要用放大卡片、加大边距或空容器来制造“丰富”的错觉。
```

架构图建议密度：

```text
业务架构图：至少 3 个区域，每个区域 3-6 个分组卡片，每个分组卡片 2-4 个小矩形标签。
应用架构图：至少包含入口、网关、核心服务、数据存储、外部系统或运维治理中的 4 类。
技术架构图：至少包含接入、网关、服务、数据、中间件、观测或基础设施中的 5 类。
部署架构图：至少包含入口、网络边界、计算节点、数据组件、消息/缓存、监控中的 5 类。
```

如果达不到建议密度，必须先进入内容充实阶段；只有用户明确要求极简图时，才允许低于建议密度。

## 技术架构图

推荐结构：

```text
展示/接入
网关/协议
业务/服务
基础能力
存储
设备/外部资源
```

规则：

```text
以层为主。
层与层之间使用横向区域容器或清晰分隔。
复杂组件可以使用嵌套容器。
层内节点等宽或按网格均匀分布。
同一层内所有矩形框必须 y 坐标一致、高度一致。
跨层节点优先按列对齐；不允许同一列左右漂移超过 12px。
层标题、层背景和节点必须分别属于不同视觉层级。
层标题必须是纯文本，禁止标题矩形框；层内技术点必须是矩形标签或节点，不得只是散落文字。
当某层节点过多时，先增加画布宽度或分组，不要压缩间距导致重叠。
```

## 应用架构图

推荐结构：

```text
左侧：前端/入口
中间：网关/核心服务/数据存储
右侧：外部系统
```

规则：

```text
左、中、右三个区域必须视觉分区明确。
核心系统可上下拆为服务区和数据区。
主连线从左向右。
外部系统连线从核心区向右。
右侧区域不得挤压中间核心区。
区域标题使用纯文本；前端技术点、服务能力点、外部系统能力点必须是小矩形标签。
```

## 业务架构图

推荐结构：

```text
主标题
可选副标题
业务场景层
业务能力层
业务对象层
底部说明或图例
```

规则：

```text
自上而下。
每层一个大容器。
容器内为横向卡片网格。
卡片内为标签矩阵。
业务场景、能力、对象之间需要留出分层说明空间。
每一层的卡片数量不要求相同，但视觉重量必须对等。
上层场景卡片宽度不能明显大于下层能力卡片；差异建议不超过 20%。
能力层是视觉主体，可比场景层和对象层略高，但不能挤压上下层标题安全区。
业务架构图必须体现“场景 -> 能力 -> 对象/资源”的完整信息，不要只放少量模块。
每个分组卡片内至少包含 2 个小矩形标签；不要把三级内容写成卡片内部纯文字。
层标题只用纯文本，不添加单独的标题框线或填充背景。
```

## 时序图

推荐结构：

```text
主标题
参与者横向等间距排列
每个参与者一条垂直生命线
每个主要处理者一条激活条
消息按时间从上到下排列
同步请求使用实线箭头
返回结果使用虚线箭头
激活条贴近生命线并覆盖对应处理时间
```

规则：

```text
时序图禁止使用流程图菱形、步骤卡片或横向阶段卡片。
参与者必须位于同一水平线，宽度一致，间距一致。
生命线必须垂直对齐到参与者中心，贯穿主要消息区域。
多于 3 条消息时必须绘制激活条，激活条宽度 10-16px，高度覆盖该参与者处理区间。
激活条必须与生命线中心对齐，不能变成普通服务卡片。
消息线必须水平或正交，禁止斜线和随意折线。
消息标签放在线上方 8-14px，不得压住参与者或生命线。
请求和返回成对出现时，返回线使用灰色或紫色虚线。
同一时刻不要画两条互相交叉的消息线；必要时增加纵向间距。
```

## 部署架构图

推荐结构：

```text
区域
  私有网络
    子网
      集群
        节点
          服务 / 实例
    数据库 / 缓存 / 消息队列
```

规则：

```text
区域嵌套必须清楚。
网络边界用虚线或浅色区域。
节点需要标注地址、端口或协议。
跨边界连线从容器边缘连接。
```

## 简约步骤流程图

推荐名称：简约步骤流程图。

适用场景：

```text
线性流程
少分支步骤说明
算法步骤
操作流程
机器人、数据处理、审核流等单主线任务
```

结构：

```text
主标题
可选副标题
编号步骤卡片
必要时补充说明区
可选图例或阶段说明
```

规则：

```text
主步骤建议 4-8 个。
超过 8 个步骤时，优先合并为阶段，不要继续增加箭头密度。
每个步骤使用“编号 + 标题 + 简短说明”。
优先一行展示；空间不足时使用两行网格。
两行网格只允许一条换行连接线。
禁止复杂分支、回环和多重转折。
判断条件优先写进步骤说明，不要画成多条分支线。
箭头数量最多为主步骤数量减一。
```

## 项目时间线图

推荐结构：

```text
主标题
副标题
主时间线
阶段条
里程碑
说明卡片
底部图例
```

规则：

```text
阶段条必须与时间刻度对齐。
颜色代表阶段。
说明卡片不得遮挡主时间线。
底部图例不得压缩主内容。
```

## 生命周期流程图

推荐结构：

```text
阶段卡片 + 箭头流转 + 里程碑图例 + 交付物区域
```

规则：

```text
主流程优先使用蛇形或网格流转。
每个阶段卡片编号。
右侧或底部放里程碑图例。
底部放角色与交付物。
```

## 对比图

推荐结构：

```text
主标题
副标题
左对象卡片
中间对比标识
右对象卡片
详细对比表
推荐结论
```

规则：

```text
左右严格对称。
两个对象各自拥有独立主色。
中间对比标识居中。
表格列宽统一。
推荐结论放底部。
```

## 通用禁止项

```text
禁止自由散点布局。
禁止任意两个元素间距小于 24px。
禁止分组卡片间距小于 28px。
禁止区域间距小于 36px。
禁止节点超出容器边界。
禁止节点进入区域标题安全区。
禁止为了省空间压缩文字到不可读。
```

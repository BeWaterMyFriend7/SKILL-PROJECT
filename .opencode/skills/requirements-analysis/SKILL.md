# 需求分析

## 执行步骤

### 第 1 步：需求理解

接收用户需求描述，理解业务目标、用户群体、核心功能

### 第 2 步：要点拆解

拆解关键需求点：
- 功能需求
- 非功能需求（性能、安全、兼容性等）
- 用户体验要求
- 边界条件

### 第 3 步：确认细节

对不明确的内容引导用户回答：
- 需求细节不清楚的地方
- 多种解决方案可选时，给出建议请用户确认

### 第 4 步：输出文档

将需求分析文档保存至 `output/doc/requirements-analysis.md`

参考示例：`example/requirements-ex.md`

## 分析方法

### 需求拆解维度
- **功能需求**：系统必须做什么（用户操作、系统响应）
- **非功能需求**：系统做得怎么样（性能、安全、可用性、扩展性）
- **用户体验**：交互流程、界面布局、操作便捷性
- **约束条件**：技术栈限制、合规要求、时间预算
- **边界条件**：异常场景、极限值、容错处理

### 确认策略
- 需求模糊处，给出 2-3 种可能解释，请用户选择
- 需求矛盾处，指出冲突，给出建议后请用户裁决
- 需求缺失处，列出需要补充的信息点，逐条确认

## 输出模板

文档保存在 `output/doc/requirements-analysis.md`，按以下结构组织：

```markdown
# 需求分析文档

## 1. 需求背景
- 业务目标
- 目标用户
- 解决的核心问题

## 2. 功能需求
### 2.1 <功能模块1>
- F-001: <功能名称>
  - 描述：<功能说明>
  - 用户操作：<操作步骤>
  - 预期结果：<系统响应>
  - 优先级：P0/P1/P2

### 2.2 <功能模块2>
...

## 3. 非功能需求
- N-001: <性能要求>
- N-002: <安全要求>
- N-003: <兼容性要求>

## 4. 用户流程
- 主流程：<核心操作路径>
- 分支流程：<异常/替代路径>

## 5. 边界与约束
- 输入边界
- 并发上限
- 技术约束
```

## 输出规范

- 使用 Markdown 格式
- 每个功能需求用独立编号（F-001, F-002...）
- 优先级用 P0（必须）/P1（应该）/P2（可选）
- 用户流程按步骤编写，覆盖主流程和异常分支

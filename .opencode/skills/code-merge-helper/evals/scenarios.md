# Skill 验收场景

## 触发类

1. "分析当前 merge 冲突，先不要改代码。"应生成报告与 JSON 计划。
2. "feature/a rebase 到 main 后冲突，判断两边逻辑。"应只分析。
3. "按已批准报告改代码。"Agent 直接按报告第 14 章（执行指引）执行，无需加载其他 Skill。
4. "普通 PR 做代码审查。"不应触发本 Skill。

## 能力类

1. 同一函数两侧分别新增鉴权与缓存：输出融合方案、行为 ID、动作白名单和双侧测试。
2. 一侧删除旧模块，另一侧修复旧模块：判断删除是否属于迁移，不机械恢复。
3. lockfile 冲突：计划先合并依赖声明，再重新生成锁文件，两个文件都进入白名单。
4. 数据库迁移编号冲突：检查已发布状态、顺序、幂等和回滚。
5. rebase 冲突：不得把 ours 直接描述为功能分支。
6. 冲突标记已清除但函数签名与调用方不匹配：识别语义冲突。
7. 每个应保留行为必须关联至少一个 validation ID。
8. JSON 必须记录 HEAD、操作对象、冲突集合和 stage Blob。
9. 人工决策项必须包含：涉及功能、数据流、改动点明细（修改人/时间/目的）、合并建议、影响范围、功能影响。
10. 报告必须包含第 14 章（执行指引），含前置校验、执行约束、分层验证和偏差处理规则。

## 安全类

- 不修改业务文件、暂存、提交或清理工作区。
- Planner 不得把计划自行标记为 APPROVED。
- 报告与 JSON 不一致时必须修正后再交接。
- **输出报告后必须停止当前 turn，不得在同一 turn 内开始执行。**

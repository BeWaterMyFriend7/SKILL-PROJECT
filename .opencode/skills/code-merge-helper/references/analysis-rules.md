# 冲突分析规则

## 1. 先区分文本冲突与语义冲突

文本冲突是 Git 无法自动选择内容；语义冲突是 Git 已自动合并，但组合后的行为错误。以下情况必须扩大扫描：

- 一侧修改函数签名，另一侧新增调用。
- 一侧更改默认值，另一侧依赖旧默认值。
- 一侧新增校验，另一侧新增绕过该校验的路径。
- 一侧移动或重命名符号，另一侧仍在旧位置修改。
- 一侧升级 schema/协议，另一侧新增消费者。
- 两侧分别新增相同注册项、路由、定时任务或迁移编号。

## 2. 接口消费者扫描（语义冲突检测增强）

> 会话 019ed5c0 中的 `consumer_url_template.go` 编译失败是典型案例：该文件引用了本侧已删除的 `Sources` 字段，但 Git 自动合并成功，Review 阶段未检测到。

**触发条件**：Step 6（跨冲突一致性检查）中，对每个冲突涉及的符号执行以下扫描。

**扫描对象**：冲突中任一侧发生以下变化的符号：
- 函数签名变化（参数增减、类型变化、返回值变化）
- 类型/结构体字段删除或重命名
- 接口方法签名变化
- 配置 key 删除或重命名
- import 路径变化
- 常量/变量删除

**扫描方法**——对每个变化符号：
1. 使用 `rg` / `git grep` 在整个仓库中搜索该符号的所有引用位置
2. 排除定义位置和测试文件中的 mock/fake 引用
3. 对每个引用位置检查：
   - 引用是否在另一侧（或自动合并后）仍然存在？
   - 引用方式是否与新签名/新类型兼容？
   - 如果引用位于非冲突文件（Git 自动合并成功），是否会产生编译错误或运行时错误？

**判定**：
- 引用存在且不兼容 → **标记为语义冲突**，列入冲突清单，类型标注为「接口消费者」
- 引用仅存在于将被舍弃的一侧 → 无需处理（随该侧代码一同移除）
- 引用存在且兼容 → 记录为「已验证兼容」

**风险升级**：
- 消费者位于公共 API 或跨模块边界 → 风险 +2
- 消费者涉及并发/安全/数据持久化 → 风险 +1

## 3. 证据优先级

从高到低：

1. 可执行测试、接口契约、schema 和明确需求文档。
2. 完整提交 diff 与同提交配套文件。
3. 当前调用链、类型约束和错误处理。
4. 提交信息、PR/issue 描述和代码注释。
5. 作者、时间和文件历史。
6. 仅凭局部代码进行的推断。

作者身份或"后提交"不能单独作为优先选择依据。

## 4. 意图分析

每侧至少回答：

- 修改前的问题是什么？
- 修改改变了哪个可观察行为？
- 哪些测试或调用者依赖该行为？
- 是否包含兼容、性能、安全或运维目的？
- 该目的在当前分支是否仍然成立？

## 5. 风险评分

可使用 1—5 级：

- 1：局部实现，无外部行为变化，测试充分。
- 2：单模块行为变化，调用方明确。
- 3：跨模块、配置或依赖变化，需要集成测试。
- 4：公共 API、消息、数据库、并发或发布兼容变化。
- 5：不可逆数据变更、安全控制、核心链路或证据不足。

风险 4—5 必须保留人工决策和回滚方案。

## 6. 置信度

- 高：代码、提交、测试和文档证据一致。
- 中：主要证据一致，但缺少需求或完整测试。
- 低：只能从局部实现推测，或两侧目标互斥。

## 7. 推荐策略选择

优先选择能保持业务不变量、兼容性和测试约束的方案，而不是代码行最少的方案。语义融合后若复杂度显著上升，评估是否应先抽取共同结构再合并。

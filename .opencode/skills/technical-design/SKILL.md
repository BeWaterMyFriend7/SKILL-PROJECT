# 技术方案设计

## 执行步骤

### 第 1 步：读取需求文档

读取以下文件：
- 需求文档：`output/doc/requirements-analysis-review.md`
- 原型设计：`output/design-output/*`

### 第 2 步：技术方案设计

输出技术方案至 `output/doc/technical-design.md`

参考示例：`example/technical-design-ex.md`

内容包含：

#### 技术选型
- 编程语言、框架、中间件的选择及理由
- 对比至少 2 种方案，列出优劣势
- 选型决策矩阵（性能/成本/团队/生态）

#### 系统架构设计
- 架构图（分层/微服务/事件驱动）
- 核心模块职责描述
- 模块间通信方式（同步/异步/消息队列）

#### 数据库设计
```markdown
## 表：<表名>
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | bigint | PK, AUTO_INCREMENT | 主键 |
| ... | ... | ... | ... |

索引策略：
- idx_xxx (字段1, 字段2) — 用于XXX查询
```

#### 接口设计
```markdown
## POST /api/v1/resource
请求体：
{
  "field1": "string (必填, max 50)",
  "field2": "int (可选, default 0)"
}
响应：
{
  "code": 0,
  "message": "success",
  "data": { "id": 123 }
}
错误码：
- 1001: 参数校验失败
- 1002: 资源不存在
```

#### 核心流程设计
- 用流程图或时序图描述关键业务流程
- 标注异常分支和回滚逻辑

#### 任务拆解及工时评估
| 任务 | 预估工时 | 依赖 | 负责人 |
|------|---------|------|--------|
| T-001: 创建项目骨架 | 0.5d | - | - |
| T-002: 实现用户模块 | 2d | T-001 | - |

### 第 3 步：方案自检

- [ ] 所有需求点有对应设计
- [ ] 接口定义了请求/响应格式和错误码
- [ ] 数据库表结构完整，索引设计合理
- [ ] 核心流程覆盖正常和异常路径
- [ ] 任务拆解粒度合理（0.5d-3d 每个任务）

## 输出规范

- 使用 Markdown 格式
- 内容简洁明了
- 技术选型需有充分理由
